EVALUATION RUBRICS
==================

Project Title: Online Gaming Website (Rockstar Games Clone)
Category: Web Development
Reference Website: https://www.rockstargames.com
Budget Tier: Medium
Complexity Level: High


========================================================================
SCORING GUIDE
========================================================================

Each rubric category is scored on a 1–5 scale:

  5 — Exceptional: Exceeds all expectations. Production-ready quality.
  4 — Proficient: Meets all requirements with minor polish opportunities.
  3 — Competent: Meets most requirements. Some gaps or rough edges.
  2 — Developing: Partially meets requirements. Significant issues present.
  1 — Inadequate: Does not meet minimum requirements. Major gaps or failures.

CATEGORY 4: TECHNICAL QUALITY (Weight: 15%)
========================================================================

Measures architecture, code quality, and engineering standards.

Score 5 — Exceptional:
  - TypeScript is used consistently across all files with strict mode enabled
  - Components are modular, reusable, and follow single-responsibility principle
  - Folder structure matches the specification exactly
  - Server components and client components are correctly distinguished (Next.js App Router)
  - API routes return proper HTTP status codes (200, 201, 400, 401, 404, 500)
  - Authentication uses secure practices (bcrypt 12 rounds, HTTP-only cookies, token expiry)
  - Database models use proper Mongoose schemas with validation
  - Environment variables are used for all secrets and configuration
  - Code is consistently formatted (Prettier) and linted (ESLint) with no errors
  - Separation of concerns: data fetching in server components/API routes, UI in client components
  - Custom hooks abstract reusable logic (useSearch, useCart, useAuth)
  - Error boundaries handle component failures gracefully

Score 4 — Proficient:
  - TypeScript is used with minor type safety gaps (1–3 files with 'any')
  - Components are well-organized with clear separation
  - API routes work correctly with proper status codes
  - Authentication is secure with minor improvements possible
  - Code is clean and readable with consistent formatting

Score 3 — Competent:
  - TypeScript is used but with frequent 'any' types or missing interfaces
  - Components are mostly organized but some are oversized or tightly coupled
  - API routes work but may not handle all error cases
  - Authentication works but has some security gaps (e.g., weak password requirements)
  - Code formatting is inconsistent in places

Score 2 — Developing:
  - Mix of TypeScript and JavaScript, or TypeScript with many errors suppressed
  - Poor component organization with duplicated code
  - API routes have limited error handling
  - Authentication has significant security issues
  - No linting or formatting configuration

Score 1 — Inadequate:
  - No TypeScript or TypeScript is misconfigured
  - Monolithic components with no separation of concerns
  - API routes are broken or missing
  - No authentication or fundamentally insecure implementation
  - Code is disorganized and difficult to understand

CATEGORY 5: PERFORMANCE (Weight: 10%)
========================================================================

Measures responsiveness, optimization, and load times.

Score 5 — Exceptional:
  - Lighthouse Performance score >= 90 on all pages
  - First Contentful Paint (FCP) <= 1.5 seconds
  - Largest Contentful Paint (LCP) <= 2.5 seconds
  - Cumulative Layout Shift (CLS) <= 0.1
  - All images use Next.js Image component with lazy loading and WebP optimization
  - JavaScript bundle is code-split per route with no single bundle exceeding 200KB gzipped
  - Font loading uses display: swap to prevent render blocking
  - API responses return within 300ms
  - Database queries use indexed fields for all search and filter operations
  - No unnecessary re-renders in React components (verified with React DevTools)

Score 4 — Proficient:
  - Lighthouse Performance score >= 80
  - LCP <= 3.0 seconds
  - Most images are optimized and lazy-loaded
  - Code splitting is implemented
  - API responses are generally fast with occasional slow queries

Score 3 — Competent:
  - Lighthouse Performance score >= 60
  - LCP <= 4.0 seconds
  - Some images are optimized, others are uncompressed or not lazy-loaded
  - Partial code splitting
  - Some API endpoints are slow (> 500ms)

Score 2 — Developing:
  - Lighthouse Performance score 40–59
  - LCP > 4.0 seconds
  - Images are not optimized (large file sizes, no lazy loading)
  - No code splitting, large JavaScript bundles
  - API responses are frequently slow

Score 1 — Inadequate:
  - Lighthouse Performance score < 40
  - Page loads take more than 6 seconds
  - Unoptimized assets throughout
  - Render-blocking resources
  - Application feels sluggish and unresponsive

CATEGORY 8: EDGE CASE HANDLING (Weight: 10%)
========================================================================

Measures robustness, error handling, and resilience.

Score 5 — Exceptional:
  - Form validation handles all edge cases:
    - Empty submissions show all field errors simultaneously
    - Excessively long input is truncated or rejected with clear feedback
    - SQL injection and XSS attempts are sanitized
    - Duplicate email registration shows a clear "email already in use" error
    - Password strength requirements enforced with real-time feedback
  - Authentication edge cases:
    - Expired verification tokens show "link expired" with a "resend" option
    - Expired password reset tokens handled gracefully
    - Concurrent session management (login on multiple devices)
    - Invalid or tampered JWT tokens return 401 without crashing
  - Store edge cases:
    - Adding quantity beyond available stock shows an error
    - Cart handles items that become unavailable (product removed/out of stock)
    - Checkout handles payment failures with clear user messaging
    - Empty cart redirects away from checkout
  - UI edge cases:
    - Images that fail to load show a styled fallback placeholder
    - Very long titles or descriptions are truncated with ellipsis and tooltip
    - Search with no results shows a helpful "no results" state with suggestions
    - Empty data states handled for all lists (no games, no articles, no orders)
  - API edge cases:
    - Malformed request bodies return 400 with descriptive error messages
    - Unauthenticated requests to protected routes return 401
    - Non-existent resources return 404
    - Server errors return 500 with a generic user-friendly message (no stack traces)
    - Rate limiting on authentication endpoints (max 5 login attempts per minute per IP)

Score 4 — Proficient:
  - Form validation handles common edge cases (empty fields, wrong format, duplicates)
  - Authentication errors are handled gracefully
  - Store handles basic edge cases (empty cart, payment failure)
  - API returns proper error codes for most scenarios
  - 1–2 minor edge cases not handled

Score 3 — Competent:
  - Basic form validation is present (required fields, email format)
  - Authentication works but edge cases may cause confusing errors
  - Store handles the happy path but some error states are unhandled
  - API returns errors but messages are not always helpful

Score 2 — Developing:
  - Form validation is minimal (only required fields)
  - Authentication errors may crash the application or show raw error messages
  - Store does not handle edge cases (e.g., empty cart goes to checkout)
  - API error handling is inconsistent

Score 1 — Inadequate:
  - No form validation
  - Authentication failures crash the application
  - No error handling in the store flow
  - API errors expose stack traces or raw database errors
